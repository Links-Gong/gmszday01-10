# Day5-Day6 DM 业务结果层

## 本层位置

DM 位于主题汇总和看板之间。它把 DWS 中的稳定汇总转成可以直接解释和展示的业务指标，同时从 DWD 形成企业销售排行。

这一层负责固化指标公式。看板只读取结果，不再自行决定快递估算、环比、同比或企业排名的计算方式。

## 步骤逻辑

### 步骤 1：定义月度指标和企业排行模型

`sql/01_create_tables.sql` 定义月度业务指标表和企业销售排行表。

月度表的粒度是“月份 + 平台”，用于 KPI 和趋势；企业表的粒度是“月份 + 平台 + 企业”，用于销售排行。先分开两种粒度，可以避免月度总量和企业明细在同一张表中重复。

### 步骤 2：把平台月汇总转换为业务 KPI

`dm_monthly_business_metrics_2026_gj` 继承 DWS 平台月的零售额、销量、店铺数、上月金额和环比，并生成物流估算指标：

```text
快递收入估算 = 零售额 × 15%
快递量估算   = 销量 × 53%
```

销售额或销量为 `NULL` 时，对应估算仍保留 `NULL`，不使用 0 伪造完整数据。

这一步先形成 2026 当前期指标，并把同比状态设为等待基期。随后再由 2025 分支补充同比，保持当前期计算与历史基期准备相互独立。

### 步骤 3：从 DWD 形成企业销售排行

`dm_enterprise_sales_rank_2026_gj` 按月份、平台和企业汇总销售额、销量和店铺数，再在每个平台内部排名。

企业名称存在时，同名企业的多个店铺合并；企业名称缺失时，使用平台和店铺 ID 生成独立的未知企业标识，避免不同未知店铺被错误合并。

企业排行直接读取 DWD，而不读取平台月 DWS，因为平台月汇总已经失去企业维度。这是 DWD 与 DM 的一条独立连续路径：

```text
DWD 店铺月明细 → 企业月汇总 → 平台内企业排名
```

### 步骤 4：接入 2025 基期并计算同比

同比把 2026 平台月指标与 `dws_platform_month_summary_2025_gj` 按相同平台和 `month_id - 100` 关联。

```text
同比增长率 = (2026 当月销售额 - 2025 同月销售额)
             / 2025 同月销售额 × 100%
```

只有当前金额和基期金额都有效，且基期不为 0 时，状态才是 `CALCULATED`。缺失基期、空基期、0 基期或空当前值都保留对应状态，并让同比值保持 `NULL`。

这样可以区分“没有同比条件”和“同比刚好为 0%”，避免错误业务解释。

### 步骤 5：确认业务指标可供看板使用

DM 校验需要证明：

- 月度指标覆盖 6 个月、4 个平台，共 24 行；
- 零售额、销量和店铺数与 DWS 平台月一致；
- 快递收入 15% 和快递量 53% 公式差异为 0；
- 同比状态、基期和同比值相互一致；
- 24 个同比均能在当前基期数据下正确计算；
- 同一月份、平台、企业不存在重复记录；
- 企业排名在平台内部按销售额正确排序。

## 本层主要表

| 表 | 业务粒度 | 看板用途 |
|---|---|---|
| `dm_monthly_business_metrics_2026_gj` | 月 + 平台 | KPI、趋势、平台分布、环比和同比 |
| `dm_enterprise_sales_rank_2026_gj` | 月 + 平台 + 企业 | 各平台企业 TOP20 |

## 与看板的连续关系

月度指标表提供已经验证的业务公式，企业排行表提供已经计算的排名。看板只根据所选月份和平台读取相应结果，从而保证页面数字与数据库结果一致，也避免在页面运行时扫描完整 DWD 或重新聚合约 1200 万行企业数据。

结果证据说明见 [results/README.md](results/README.md)。
