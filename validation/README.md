# Day9 全链路联调与验证报告

## 本阶段位置

Day9 位于数据开发和成果提交之间。它不重建已经通过的表，也不改变业务数据，而是沿着 ODS、DWD、DWS、DM 到看板的路径逐层验证。

验证的核心不是“查询能返回结果”，而是证明每一层的输出都能由上一层解释，并且能够作为下一层的可靠输入。

验证日期：2026-07-15。验证范围包括 `202501-202506` 同比基期、`202601-202606` 主分析链路和 Streamlit 看板。

## 验证与关系

### 1：确认装载范围完整

先检查 2025 和 2026 的装载审计，确认每年都有 6 个月、4 个平台、24 个成功批次。

这一步证明预期源表已经进入 DWD。如果批次不完整，后续的趋势、环比、同比和平台对比都没有完整时间范围，因此不能继续做总量对账。

### 2：解释原始行数到 DWD 行数的变化

DWD 粒度是 `month_id + platform_id + shop_id`，不能要求 DWD 行数直接等于原始行数。正式核对关系是：

```text
原始行数 - 空店铺 ID - 去重或聚合差异 = DWD 目标行数
```

这一步证明减少的行数来自明确的过滤、去重和聚合，而不是装载丢失。同时检查唯一键重复为 0，确保后续 DWS 不会重复累计店铺。

### 3：确认 DWS 三种主题与 DWD 对平

平台月汇总与 DWD 比较金额、销量和店铺数；全国区域汇总与平台月汇总比较；完整类目汇总与平台月汇总比较。

这一顺序证明同一份 DWD 经过不同维度汇总后仍保持相同总量。DWS 对账通过后，平台月结果才能进入 DM，区域和类目结果才能进入看板。

### 4：确认 DM 公式和同比正确

月度 DM 与平台月 DWS 对账，并复算快递收入 15%、快递量 53%、环比和同比。

同比还要确认 2026 当前月准确关联到 2025 同平台、同月份基期。缺失或 0 基期必须保持 `NULL`，不能伪造为同比 0%。

企业排行只检查总行数、唯一性、排名范围和代表性样本，不导出完整 1200 万行作为日常验证材料。

### 5：确认看板能够消费结果表

看板验证包括服务健康、主要组件加载、月份和平台筛选、省市联动、企业排行查询性能、同比状态、空结果状态和响应式布局。

这一步把数据库正确性和页面可用性连接起来。只有页面使用了正确结果表，并能把筛选条件传递到查询，整个项目才算形成闭环。

### 6：形成统一 PASS/FAIL 结论

数据库检查和看板检查最终汇总到统一结果中。关键检查全部 `PASS`，表示项目可以进入 Day10 成果整理；任一关键检查失败，都应回到对应上游层修复并重新验证受影响部分。

## 当前总体结论

**评估结果：可以提交，需保留数据质量说明。**

- 数据库全链路 21 项检查全部 `PASS`；
- 看板 13 项服务及组件检查全部 `PASS`；

## 同比手工复算

抽查 `202601 + SMT`：

```text
2026 销售额 = 7,457,153,513.54
2025 同期   = 8,815,040,918.60
同比        = (7,457,153,513.54 - 8,815,040,918.60)
              / 8,815,040,918.60 × 100
            = -15.4042%
```

SQL 结果为 `-15.4042%`。它证明 2025 基期已经沿着 DWD、DWS 进入 2026 DM，同比关联和公式一致。

## 联调中解决的问题

### 1. 省市重复和非法选项

“云南/云南省、上海/上海市、昆明/昆明市”等别名在看板中统一为标准名称；`Office Products`、`国际邮编` 等非行政区值不再进入筛选项。

筛选标准名称时仍匹配全部原始别名，因此规范显示不会造成销售额漏计。

### 2. 企业排行查询超时

原页面逻辑会重新聚合约 1200 万行企业数据，存在 MySQL 连接中断风险。当前页面改为读取 DM 已计算的每月、每平台 TOP20。

这项修复把大规模计算留在 DM，把轻量结果交给看板，恢复了层级职责和页面响应速度。

### 3. 指标卡长数字截断

指标值字号和换行规则已经调整，长金额和销量不再被省略。该修复只影响展示，不改变 DM 指标值。
