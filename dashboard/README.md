# Day7-Day8 Streamlit 数据看板

## 本层位置

看板是数据链路的展示层。它读取已经通过校验的 DWS 和 DM 结果，把月度 KPI、趋势、平台、类目、企业和地区数据放到同一页面中。

看板不修改数据库，不重新清洗源数据，也不在页面运行时重新定义指标公式。页面中的数字应当能够回溯到 DWS 或 DM，数据问题应返回上游修正，而不是只在界面中隐藏。

## 上游依赖

| 页面内容 | 主要数据源 | 上游职责 |
|---|---|---|
| KPI、物流估算、环比、同比 | `dm_monthly_business_metrics_2026_gj` | 提供已验证的业务公式 |
| 企业销售排行 | `dm_enterprise_sales_rank_2026_gj` | 提供每月、每平台企业排名 |
| 类目 TOP10 | `dws_category_month_summary_2026_gj` | 提供完整类目汇总，支持所选平台重排 |
| 省市筛选、区县排行、地图 | `dws_region_month_summary_2026_gj` | 提供不同 `region_level` 的区域汇总 |

只有这些上游表完成刷新和校验后，看板结果才具有业务意义。

## 页面逻辑

### 步骤 1：确定当前分析范围

页面先读取可用月份和平台，形成全局筛选条件。

月份采用单点滑块。滑块停在哪个月，KPI、平台、类目、企业和区域查询就只读取该月，不再把起止月份之间的数据累加。默认月份为当前数据范围中的最后一个月。

平台支持多选，所选平台影响全部主要结果。这样同一个页面可以查看全平台，也可以缩小到一个或多个平台。

### 步骤 2：展示核心业务结果

页面首先展示零售额、销量、快递收入估算、快递量估算和活跃平台店铺数。

这些指标直接来自 DM。顶部零售额环比会单独读取上月作为基期，因此即使月份筛选只选一个月，仍可以正确显示环比。

### 步骤 3：解释平台和类目构成

平台分布展示所选月份、所选平台的销售额构成；类目 TOP10 从完整类目汇总中重新聚合和排序。

“未分类”是上游类目缺失的真实汇总结果，当前不会在看板中擅自删除、平滑或分配给其他类目。

### 步骤 4：提供企业排行

企业排行读取 DM 已计算的各平台 TOP20。选择多个平台时，页面分别展示这些平台的排名，不在运行时重新扫描完整企业排行表。

这一设计保证页面响应速度稳定，也保持排名口径与 DM 一致。当前排行是平台内排行，不是多个平台合并后的全平台排行。

### 步骤 5：提供地区下钻

省份和城市筛选作用于区域选项、区县 TOP15 和地图。页面会把“云南/云南省”“昆明/昆明市”等同义行政区合并为标准显示名称，同时用全部原始别名查询，避免漏计。

非行政区文本不进入筛选列表，但底层数据不会被删除。地区结果只代表能够识别到对应行政区的记录，不能替代全国总量。

### 步骤 6：展示数据状态和空结果

当销售额有效店铺覆盖不足时，页面显示覆盖率提醒；当基期缺失或为 0 时，同比显示无可用同期基期；当筛选没有结果时，页面显示明确空状态。

这些提示把 DWD 和 DM 中的质量状态传递到展示层，防止用户把缺失数据误解为真实 0。

## 页面组成

- 月度零售额和销量指标；
- 物流收入与物流量指标；
- 平台销售额分布；
- 类目销售额 TOP10；
- 区县销售额 TOP15；
- 各平台企业 TOP20；
- 平台月度环比和同比；
- 省域销售分布地图。

## 项目文件职责

| 文件 | 作用 |
|---|---|
| `app.py` | 页面结构、筛选状态、数据库查询和图表展示 |
| `requirements.txt` | 声明看板运行所需 Python 依赖 |
| `.streamlit/secrets.toml` | 保存本地数据库连接信息，不进入版本库 |
| `results/` | 保存页面验收截图和检查结论 |

## 与 Day9 的连续关系

Day9 不只检查服务能否打开，还要验证筛选器能否改变数据、主要图表能否读取真实结果、企业排行是否避免全表扫描、地区别名是否合并，以及窄屏下是否可用。

只有页面结果能回溯到 DWS 和 DM，并且交互没有运行错误，看板才算完成。验收证据说明见 [results/README.md](results/README.md)。
