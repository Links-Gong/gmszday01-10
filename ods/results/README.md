# ODS 结果证据

## 目录作用

本目录保存 Day1-Day2 的认识结果和质量证据。这里的文件不是下游计算输入，而是用来说明 DWD 的字段映射和清洗规则为什么这样制定。

## Day1：结构认识证据

### 平台字段描述

四个平台的 `DESCRIBE` 结果记录各源表的字段名称和类型，用于识别平台之间的同义字段及结构差异。

### 字段对比表

`field_comparison.xlsx` 把店铺、企业、地区、销量、销售额和类目等核心字段放在同一张对照表中。它把“认表”结论传递给 DWD 的统一字段设计。

### 源表预检结果

预检结果分别证明：24 张源表是否齐全、必需字段是否齐全、跨月结构是否一致、维表字段是否满足关联需要。

这些结果共同决定数据能否从 ODS 进入 DWD。缺表、缺字段或结构漂移不能仅记录后忽略，必须先解决或明确调整装载口径。

## Day2：数据质量摸底证据

| 结果主题 | 说明的问题 | 对 Day3 的影响 |
|---|---|---|
| `category_new缺失` | 各平台标准类目覆盖情况 | 决定类目是保留、回退还是标记为未分类 |
| `dim与shop_id匹配` | 店铺与企业维表的匹配覆盖 | 决定哪些企业和地址可由维表补充 |
| `sales =0` | 源数据中的真实 0 分布 | 要求 DWD 将真实 0 与缺失值分开 |
| `改名店铺` | 同一店铺 ID 是否出现不同企业名称 | 要求统一粒度后保留稳定的店铺识别方式 |
| `省份缺失` | 行政区字段的可用程度 | 决定区域分析覆盖范围和后续质量说明 |

## 结果之间的连续性

字段描述先确定“有哪些字段”，字段对比再确定“哪些字段含义相同”，质量摸底最后确定“这些字段是否可信以及应如何处理”。三部分共同形成 DWD 装载规则，不能只依据其中一份结果直接合并平台数据。

## 完成标志

本目录能够回答以下问题时，ODS 阶段才算完成：

- 24 张主分析期源表是否完整；
- 四个平台的核心业务字段如何对应；
- 哪些数据问题需要在 DWD 中处理；
- 哪些缺失和异常必须保留并进入质量统计。
