# Day1-Day2 ODS 与源数据认识

## 本层位置

ODS 阶段位于整条链路最前端。它不修改 `ec_cross_border` 中的源表，也不生成业务指标，而是回答两个问题：源数据能否进入后续处理，以及后续需要按什么规则处理。

本层的结论直接交给 DWD。DWD 中的字段映射、空值处理、去重方式和维表关联规则，都必须能追溯到这里发现的数据特征。

## 步骤逻辑

### 步骤 1：识别四个平台的数据结构

确认 SMT、Amazon、Alibaba、Ozon 的月表命名规律和字段含义，重点识别店铺 ID、企业名称、地区、销量、销售额和类目字段。

这一步解决“相同业务含义在不同平台叫什么”的问题。产出的字段对照关系是后续把四个平台合并到统一 DWD 字段的基础。

### 步骤 2：确认 24 张主分析期源表完整

检查 202601-202606 每个月、每个平台的源表是否存在，共应覆盖 24 个批次。

这一步防止在缺少某个月或某个平台时继续装载，因为缺表会直接造成后续趋势、环比和平台对比不完整。

### 步骤 3：确认关键字段可用且跨月一致

检查每个平台的必需字段是否存在，并以 202601 为基准比较其余月份是否发生字段缺失或类型漂移。同时确认企业维表具备 Day3 关联所需字段。

这一步保证同一平台的六个月能够使用同一套装载逻辑，也保证 DWD 可以安全关联企业维表。

### 步骤 4：识别数据质量问题

对空店铺 ID、销售额或销量为 0、类目缺失、省份缺失、同店铺改名和维表匹配情况进行摸底。

这一步不是为了直接填补数据，而是区分真实 0、缺失值和异常值，并判断哪些字段可以用可靠来源补充。结论转化为 Day3 的清洗和审计规则。

## 本层产出

- 四个平台字段对照和结构说明；
- 24 张源表的存在性结论；
- 必需字段和跨月结构一致性结论；
- 数据质量风险清单；
- DWD 字段映射、清洗和维表关联的依据。

## 与 DWD 的连续关系

只有当源表齐全、关键字段齐全、结构漂移可控时，DWD 才能开始批次装载。质量摸底中发现的问题不会在 ODS 被隐藏，而会在 DWD 中以过滤、保留 `NULL`、去重、维表补充或审计计数的方式明确处理。

对应的只读预检逻辑集中在 `sql/01_preflight_source_tables.sql`，结果证据说明见 [results/README.md](results/README.md)。

- 已形成能够指导 DWD 的字段和质量规则。
